1.3.3并发容器类list_set_queue

并发容器类list_set_queue

# 一、List

CopyOnWriteArraylist容器即写时复制的容器

和ArrayList比较，优点是并发安全，缺点有两个：

  1. 多了内存占用：写数据是copy一份完整的数据，单独进行操作。占用双份内存。
  2. 数据一致性：数据写完之后，其他线程不一定是马上读取到最新内容。

二、Set集合

Set和List重要区别：不重复

实现 原理 特点
HashSet 基于HashMap实现 非线程安全
CopyOnWriteArraySet 基于CopyOnWriteArrayList 线程安全
ConcurrentSkipListSet 基于ConcurrentSkipListMap 线程安全，有序，查询快

PS：添加对象时，需重写该对象的equals()和hashCode()

三、Queue API

Queue-队列数据结构的实现。分为阻塞队列和非阻塞队列。下列的蓝色区块，为阻塞队列特有方法。

方法 作用 描述
add 增加一个元素 如果队列已满，则抛出一个IllegalStateException异常
remove 移除并返回队列头部的元素 如果队列为空，则抛出一个NoSuchElementException异常
element 返回队列头部的元素 如果队列为空，则抛出一个NoSuchElementException异常
offer 添加一个元素并返回true 如果队列已满，则返回false
poll 移除并返回队列头部的元素 如果队列为空，则返回null
peek 返回队列头部的元素 如果队列为空，则返回null
put 添加一个元素 如果队列满，则阻塞
take 移除并返回队列头部的元素 如果队列为空，则阻塞

并发容器类map

一、JDK源码学习方法更重要

逻辑思维能力是梳理学习方法的基础。养成线性思维：两个或者多个概念，像一条线串起来。

  • 演绎推导法

    示例：因果推理。因为Java中网络编程只提供了BIO和NIO两种方法，所以一切框架中，涉及到网络处理的，都 可以用这两个知识点去探究原理。

  • 归纳总结法

    示例：可能正确的猜想。线上10台服务器，有三台总是每天会自动重启，收集相关信息后，发现是运维在修改监控系统配置的时候，漏掉了提高这三台机器的重启阀值。

  • 类比法

    集群概念就好像是马在拉车，一匹马拉不动的时候，就使用多匹马去拉。

    分布式的概念，就像是理发的过程中，洗头发和剪头发是不同的人负责的。

二、推理HashMap的实现

  1. 数据要存储

    涉及到数据结构：数组、链表、栈、树、队列

  2. 数组的插入和查找

    1. 顺序查找：插入时按先后顺序插入，查找时轮询扫描进行比对。
    2. 二分查找：插入时进行排序；查找时将n个元素分成大致相等的两部分，减少复杂度。
    3. 分块查找：分块查找是二分查找和顺序查找的一种改进。
    4. 哈希表：对元素的关键信息进行hash计算，求出下标后直接插入或查找。常用的实现是除留余数法。

  3. 哈希冲突，数组位置已存在值

    hash(key2)=hash(key1)。链地址伐；ReHash1(key2)再次计算hash；

  4. 合理控制数组和链表的长度

    动态扩容resize()

三、HashMap-非线程安全

PS：实现线程安全的三种方法：

  1. sync关键字
  2. lock
  3. cas机制

PS：扩容是什么？

  1. 计算一个新的大小
  2. 构建一个newTable
  3. 遍历oldTable添加到newTable

四、ConcurrentHashMap1.7-线程安全

HashTable太过于粗暴，推荐使用ConcurrentHashMap，包一层segment确保线程安全

五、ConcurrentSkipListMap

特点：有序链表实现，无锁实现；value不能为空；层级越高跳跃性越大，数据越少，查询理论变快。

新的node是否抽出来作为index，随机决定；

index对应的level由随机数决定。（随机数比特位连续为1的数量）

每层的元素，headIndex固定为所有node中最小的；

查找数据时，按照从上到下，从左往右的顺序查找

时间复杂度O(log n)，空间复杂度O(n)。

空间换时间，数据库索引类似的概念，skiplist在很多开源组件中有使用（level DB，Redis）

Starter原理解密

  • Starter的作用：

    • starter引入相关的jar包
    • starter自动完成bean配置

  • 疑问1：这些bean的依赖关系是如何自动处理的？

    • 首先明确一下spring配置bean的方式有几种？

      • xml配置
      • 注解
      • 基于Java代码的方式（@Configuration+@Bean）

    • 如：如果我们没有配置datasource，那mybatis会被加载吗？

    • 自动配置–条件依赖注解：（通过这些条件依赖来说明定义，SpringBoot容器可以自动Bean的配置）

      • @ConditionalOnClass、@ConditionalOnMissingClass

      • @ConditionalOnBean、@ConditionalOnMissingBean

      • @ConditionalOnProperty

      • @ConditionalOnResource

      • @ConditionalOnWebApplication、@ConditionOnNotWebApplication

      • @ConditionalOnExpression

- @AutoConfigureAfter  @AutoConfigureBefore  @AutoConfigureOrder指定顺序

  • 疑问2：这些bean配置需要的参数是如何规定并获取的？

    • 如：配置datasource需要数据库连接参数，mybatis也有自己的配置参数，这些参数在编写的时候如果没有提示怎么办？
    • 找到jar包，找到META-INF目录，里面有个spring-configuration-metadata.json文件，这个文件就是记录配置参数的

  • 疑问3：这些bean配置是如何被发现并加载的？

    • SpringApplication构造方法里有一个setInitializers()初始化器和一个setListeners()监听器，它们会去加载spring-configuration-metadata.json里的对应参数，具体可以看getSpringFactoriesInstances()

  • 思考：如果我们开发的一些通用组件，被很多项目使用，我们是不是可以制作一个starter，来简化使用？

开发自己的Spring-Boot-Starter

  • Starter是一个集成接合器，完成两件事

    • 引入相关的jar
    • 自动配置

  • Spring boot规范：

    • start.jar完成引入相关的jar

    • autoConfigure.jar完成自动配置

      也可以只用一个jar包

  • Starter命名规范

    • Spring提供的starter：
      • spring-boot-starter-xxx-x.y.z.jar
      • spring-boot-xxx-autoconfigure-x.y.z.jar
    • 第三方的starter：
      • xxx-spring-boot-starter-x.y.z.jar

  • 动手制作一个第3方lib的starter

    • 一：准备第3方的jar
    • 二：制作starter
      • 1建工程
      • 2引入spring-boot-start、spring-boot-autoconfigure、第三方jar
      • 3如需要生成配置元信息，加入spring-boot-configuration-processor依赖
      • 4编写自动配置类
      • 5配置发现配置文件：META-INF/spring.factories
      • 6供其他项目使用

  • 我的疑问：上面这个过程跟maven打包成一个jar包是一样的吗？不一样的话有什么区别？打包完了之后怎么使用？

使用Hexo创建自己的博客过程中遇到的一些坑

发表于 2020-01-11 更新于 2020-12-19

Hexo的详细教程可以参考中文官方文档：https://hexo.io/zh-cn/docs/，因为是翻译过来的，所以有些表述其实不是很清晰。以下是个人在使用过程中遇到的一些小坑，特此记录分享

第一个坑：Hexo安装完成后需要配置环境变量

Hexo安装完成后需要配置环境变量才能在dos窗口使用hexo命令，简单配置一下（跟配置jdk环境变量类似不懂问度娘）：

  • 配置环境变量：E:\Github\hexo\node_modules\.bin
  • 当然，你的路径可能和我的不同。但是后面的一定是\hexo\node_modules\.bin
  • 然后，重新打开一个命令行就可以执行hexo -v的命令查看hexo版本

第二个坑：Git安装可以选择自动配置环境变量

安装的时候有个步骤选择“系统自带CMD：使用windows系统的命令行工具”，即选择第二个“use git from the windows command prompt”，就会自动帮你配置好环境变量不需要自己配置。

第三个坑：初始化根目录

安装 Hexo 完成后，请执行下列命令，Hexo 将会在指定文件夹中新建所需要的文件

第四个坑：根目录下有个themes目录

这个目录是专门用来放各种主题的，一般都是通过git下载，所以这里需要切换到themes目录下再进行下载，主题的选择需要到hexo主目录下的config.yml去设置，hexo主目录就是你执行命令hexo init <folder>得到的那个folder目录

第五个坑：配置文件有两种

我的理解是有站点配置文件和主题配置文件两种：

  • 站点配置文件在主目录下config.yml，主要是有关站点的一些配置，如网站标题、副标题、网站主题选择等；

  • 每种主题一般都有对应的配置文件，可以到themes选择对应的主题找到_config.yml进行配置，比如一种主题可能有多种风格，风格的配置就是在这里配。

第六个坑：标签、关于、分类设置

以标签为例：

①执行命令：hexo new page “tags”，执行完之后可以到主目录的sources里看到tags目录

②编辑tags目录下的index.md，编辑内容更改为：

title: tags
date: yyyy-mm-dd 13:49:35 #title和date会自动生成，这里注意要多设置一个type，标签tags和分类categories需要设置一下，关于about不需要
type: “tags”

③注意这里要设置type而已一定是tags，说设置layout:”tags”都是错了（忘记是在哪里看到的教程，坑！）
④到主题配置文件设置下菜单，启动需要的菜单
menu:
home: / || home
about: /about/ || user
tags: /tags/ || tags
categories: /categories/ || th
archives: /archives/ || archive

第七个坑：部署到github

这里我就简单说一下部署的主要流程：

  1. 到github创建仓库R
  2. 到hexo目录下的_config.yml设置deploy信息
  3. 将hexo代码上传到R仓库
  4. 部署到Github

具体可参考：https://www.cnblogs.com/admol/p/admol.html
